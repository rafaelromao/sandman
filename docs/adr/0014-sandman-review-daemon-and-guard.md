# ADR-0014: Sandman Review - Daemon-Monitored PR Reviews

## Status

proposed

## Context

Sandman needs a PR review workflow: when someone posts `/sandman review [focus]` on a GitHub PR, a review agent should read the diff and post findings as a comment. This introduces a new command and a coordination problem.

### Three design questions

1. **One-shot vs daemon**: Should `sandman review` scan PRs once and exit, or run as a background daemon that polls for new commands?
2. **Who posts the review**: Should the sandman daemon run the agent and post its output, or should the agent itself post the comment?
3. **How to ensure the daemon is running**: The default `review_command` config value drives the pr-review skill's PR comment. When it contains `/sandman`, the agent posts a comment that the daemon must intercept. If the daemon isn't running, that comment hangs. How do we prevent this?

## Decision

### Two modes: daemon (default) and one-shot

`sandman review` defaults to **daemon mode** - it polls the repo every 60 seconds for new PRs with `/sandman review` comments. PR numbers passed as positional arguments run in **one-shot mode** for manual or CI-driven review invocation. One-shot mode accepts bare numbers, closed ranges (`N:M`), and unbounded ranges (`N:`, `:M`).

Rationale: the daemon enables the full automated workflow (AFK agent finishes work, posts `/sandman review`, daemon picks it up). The one-shot mode covers ad-hoc cases without running a background process.

### Agent posts the review comment

The review agent is responsible for posting its findings as a PR comment via `gh pr comment <N>`. The daemon only launches the agent - it does not parse or relay output.

Rationale: keeps the daemon simple and lets the prompt control the posting format. The daemon does not need to understand agent output.

### Control-socket guard

When the configured `review_command` contains `/sandman`, the `sandman run`, `sandman run --continue`, and `--auto` selection commands check for a control socket at `.sandman/review.sock` **before** executing. If the socket is missing, the command fails with a clear message.

This prevents the following failure mode:
1. User runs `sandman run` with default config
2. Agent finishes, posts `/sandman review` on the PR per the pr-review skill
3. Nobody is watching - review never happens

The socket check adds a forward dependency (review daemon must be running before you can run batches with the default review command), but it is a **fail-fast** dependency: the user sees the error immediately, not after a batch completes.

### Why a socket over alternatives

- **PID file**: Stale PID files survive crashes, leading to false positives (run thinks daemon is alive when it's dead).
- **Event log**: Requires reading and parsing the JSONL log, which is slower and less reliable than a single `os.Stat`.
- **Port check**: Requires binding a TCP port - overkill for a local workflow tool.

The socket (`net.Listen("unix", ...)`) is already an established pattern in the codebase (see `daemon.ControlSocket` and `attach`) and gives us a natural attach endpoint for future `sandman review --attach` functionality.

## Consequences

### Positive

- Fail-fast guard prevents the "orphaned review comment" failure mode.
- Two modes cover both automated and ad-hoc workflows.
- One-shot mode shares all prompt and agent infrastructure with daemon mode - no duplication.
- Socket-based guard reuses existing, tested infrastructure.
- Users can opt out of the guard entirely by setting `review_command` to something that does not contain `/sandman` (e.g. `/oc review` for backward compat).

### Negative

- Forward dependency: `sandman run` fails if `review_command` contains `/sandman` and no daemon is running. Users must either override `review_command` or start the daemon.
- Daemon consumes a terminal or must be started as a background job.
- The socket check adds a filesystem operation to the startup path of `run`/`continue`.

### Neutral

- The default `review_command` changes from `/oc review` to `/sandman review`.
- The one-shot mode is a thin wrapper around `BatchRunner.RunBatch` - no new execution infrastructure.

## Self-posted comment filter (issue #1648, refreshed by #1756, #1757, #1759)

The review workflow is a closed loop: the implementor agent posts `/sandman review ...` on a PR, the daemon picks it up, launches a reviewer agent, the reviewer agent posts the actual review. The reviewer agent's review-comment body is generated by the agent and posted through the host user's `gh` config, so the daemon cannot tell "the bot is asking for a review" from "the bot just reviewed" by author alone. If the bot's review-comment body ever contains the trigger substring (a self-review summary quoting the request, a future skill variant, etc.), the daemon will re-launch the review and the bot will loop on itself.

The bot's own review run log — `.sandman/batches/<batch-id>/runs/<run-id>/run.log` — is the daemon's authoritative source for which bodies the bot posted. The daemon greps that log to discover its own bot's review-comment posts and records them into `SelfPostStore`, then drops any matching comment before parsing it for the trigger substring. This breaks the self-loop without depending on the skill or the LLM agent to maintain the daemon's state.

### Decision

Track the SHA-256 hash of the bot's review-body posts, scoped per PR number, and ignore any comment whose `(PR, body)` matches a recorded entry.

### Storage and lookup

- **On-disk shape**: `<.sandman>/reviews/self-posted.json` is a single JSON object whose keys are the composite `pr-<N>-<sha>` string (where `<N>` is the PR number and `<sha>` is hex(`sha256(normalized body))`); the values carry the entry's diagnostic metadata (`sha256`, `pr_number`, `run_id`, `posted_at`). The on-disk file is rewritten atomically via temp-file + `os.Rename` on every `Record`. A missing file is acceptable — the loader does not synthesise one; the next `Record` creates it.
- **In-memory shape**: `map[int]map[string]selfPostEntry` — outer key is the PR number, inner key is the hex sha256. Each per-tick `IsSelfPosted` lookup is two map accesses.
- **Dedup key**: `(prNumber, sha256(body))`. The body hash alone is not sufficient — the bot's review body on PR A is a different entry from the same body on PR B. **per-PR scoping closes the cross-PR poisoning failure mode observed on PR 1752**: a trigger hash recorded against PR A no longer silently drops the same body on PR B. The regression test pinning this is `TestSelfPostStore_NoCrossContamination_OnRealisticPoisonShape`. The per-PR shape of the in-memory map (`map[int]map[string]selfPostEntry`) and of the on-disk JSON object (`pr-<N>-<sha>` keys) is what enforces this scoping.

### Lookup contract

`SelfPostStore.IsSelfPosted(prNumber int, body string) bool` is the sole public read API. Callers ask "would this `(PR, body)` pair be a self-post?" — nothing else. The check is constant-time over the in-memory map; the on-disk file is never consulted on the read path.

### Write path (daemon-owned)

`Daemon.processPR` is the sole authoritative record site of bot-posted bodies in `SelfPostStore` — the daemon's review path owns the store end-to-end. The actual `Record` call lives inside `Daemon.promotePendingComment`, the per-pending-entry helper called by `promotePendingReviews`, which `processPR`'s tick launches at the start of every tick. The write path is gated by the bot's own review run log:

1. `processPR` launches the reviewer agent; the agent's `run.log` (under `.sandman/batches/<batch-id>/runs/<run-id>/run.log`) accumulates every shell invocation the agent runs.
2. On every tick, `promotePendingReviews` calls `extractBodiesFromLog(runLogPath)` once per unique log file (per-tick cache) to recover every body the bot posted via `gh pr comment <N> --body <body>` (or `--body-file`).
3. `promotePendingComment` string-equals each post-`since` PR comment against the recovered bot bodies. A match is recorded via `SelfPostStore.Record(prNumber, body, runID)`; a non-match is treated as a human reply and not recorded. Recording only confirmed bot bodies preserves the #1722 contract that defensive observation of every observed comment would poison the trigger hash.
4. On the next tick, the same `(PR, body)` is already in the store and `processPR`'s `IsSelfPosted(prNumber, comment.Body)` call — placed before `ParseTrigger` per #1702 — drops it before its body is parsed for the trigger substring. This breaks the self-loop in PR #1671, where the bot's review body contained the literal `/sandman review` substring in its `## Previous review progress` section.

The `pr-review` skill no longer maintains `.sandman/reviews/self-posted.json` from any wrapper; the LLM agent is not in the loop on the store's write path. The skill-side responsibility-boundary violation that produced cross-PR poisoning on PR 1752 is gone.

### Loader behavior (issue #1756)

`NewSelfPostStore` is greenfield. Any pre-existing `self-posted.json` at startup is atomically renamed to `self-posted.json.ignore-<ts>.bak` exactly once, the in-memory store starts empty (a one-line info log records the rename), and the loader returns no error. The file is rewritten via the existing atomic temp-file + rename path on the next `Record`. The renaming is irrecoverable from the loader's perspective — operators who want to retain the archived bodies for forensics read the `.bak`; the new store starts clean. This sweep is the only safe recovery for a key-shape change, because silently merging legacy single-body-hash entries into the new `(prNumber, sha)` space would either drop them (silent data loss) or mis-attribute them to an arbitrary PR (silent mis-scoping).

A missing file at startup is also acceptable: no rename to perform, store starts empty.

### Discovery via run-log grep (issue #1759)

The bot's own review run log is the daemon's authoritative source for which bodies the bot posted. The helper `extractBodiesFromLog(runLogPath)` in `internal/review/agentlog.go` greps the per-run run.log for `gh pr comment <N> --body <body>` invocations and returns every body the bot posted, in chronological order. The parse is robust against:

- **Quote shapes**: `--body '...'` (single-quoted) and `--body "..."` (double-quoted). Bytes are passed through verbatim — bash single quotes do not interpret escapes, and the daemon does not try to second-guess the body.
- **`--body-file`**: absolute paths are used as-is; relative paths are joined against the run-log directory. A missing `--body-file` path returns an explicit error and the caller treats it as "no bodies extracted for this cycle" so the next tick retries.
- **Shell wrappers**: the agent shells out from its worktree, so the log line is typically `cd <worktree> && gh pr comment <N> --repo <owner/name> --body "..."`. The parser scans for `gh pr comment` and the `--body`/`--body-file` flag positions independently, so the `cd` and `--repo` are tolerated without special-casing.
- **Multiline bodies**: the log writer prefixes every physical line with `[<runID>] HH:MM:SS ` and drops the `$` shell prompt for continuation lines (which carry only the `HH:MM:SS ` prefix). The parser joins continuation lines into one body by detecting the timestamp-only prefix.

`pendingReviewEntry` carries the run-log path (default: `filepath.Dir(reviewState) + "/run.log"`, applied in `processPR`'s launch goroutine and in `loadPendingReviews` for entries rehydrated from disk). On every tick, `promotePendingReviews` greps each unique `runLogPath` exactly once (per-tick cache) and threads the bodies into `promotePendingComment`.

Implementation-run logs (`Kind: "issue"`) are not sources for self-post recording. The check is by log type — the helper reads the sibling `run.json` and returns no bodies when its `Kind` is not `"review"`. This is defense-in-depth on top of the caller's `batchindex.KindReview` filter (loadSeenCache / loadPendingReviews), so an implementation-run log never feeds self-post recording even when the caller misclassifies.

Quotes that are never closed return an explicit error mentioning the line number and a nil slice — partial bodies are not returned alongside the error. The caller treats an error as "no bodies extracted for this cycle" and retries on the next tick. A missing log file is a no-op (`nil, nil`); the daemon may invoke the helper before the agent has written the first log line.

### Bounded-retry grace (issue #1759)

Pre-#1759, the bounded-retry escape (3 cycles × 30s = 90s) marked a pending review as `failure` when no post-`since` PR comment had arrived. The bot's typical post latency is sometimes longer than 90s, so a successful review was wrongly labelled as a failure. The new bounded-retry grace: when the cycle counter reaches `pendingMaxCycles`, the daemon performs one final `extractBodiesFromLog(runLogPath)`. If any body is found, the entry settles as `success` and the bodies are recorded into `SelfPostStore`. Only an empty log (the bot truly failed) triggers the failure escape.

The grep is cheap — a few kilobytes of run.log per tick, parsed once per unique `runLogPath` per tick (per-tick cache) — much cheaper than the agent's typical post latency. The grace path closes the "bot was slower than 90s" race without giving up the bounded-retry escape that prevents silent retry forever.

### Ownership note (issue #1757)

The earlier design split the SelfPostStore's record site between the `pr-review` SKILL.md Step 4b wrapper (primary, ran inside the LLM agent after `gh pr comment` returned) and `Daemon.promotePendingComment` (defensive observation in the daemon). That split was the structural source of the cross-PR poisoning observed on PR 1752: the skill-side wrapper required the LLM to compute a SHA-256 of the body it just posted and write it into the daemon's state file, putting a responsibility-boundary violation in place where the daemon relied on an LLM reading its instructions correctly to maintain its own state. As of issue #1757, the skill no longer maintains `.sandman/reviews/self-posted.json` from any wrapper; `Daemon.processPR` is the sole authoritative record of bot-posted bodies — every non-trigger comment it observes is recorded, and trigger bodies are deliberately skipped to keep future trigger lookups live. The review prompt's hard rule that forbids emitting the literal `{{REVIEW_COMMAND}}` substring in the review body remains as defence-in-depth, but it is no longer load-bearing — the daemon-side filter breaks the self-loop regardless of whether the bot ever emits the substring. Issue #1759 then narrowed the actual `Record` call to live inside `promotePendingComment`, gated by the bot's own review run log, so the daemon can attribute a body to its bot without trusting the LLM or relying on `ListPRComments`'s eventual consistency.

### Why not GitHub auth?

Both the implementor and reviewer agents share the host user's `gh` credentials, so `comment.author.login` is the same value for both. Author identity cannot distinguish "the bot is asking" from "the bot just answered". Hash tracking is identity-agnostic and survives a future change in which user the agents run as.
