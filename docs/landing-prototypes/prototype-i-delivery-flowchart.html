The handoff contract

Sandman starts where a well-described issue becomes executable.

Sandman does not ask you to adopt a planning religion. It asks for a GitHub issue clear enough to delegate. The upstream process is yours. The AFK implementation loop is Sandman's.

Agent-ready issues are SMART enough to run unattended.

SMART is the minimum viable contract between a human and an AFK agent.

SSpecific: the behavior and scope are concrete.
MMeasurable: acceptance criteria or checks define done.
AAchievable: small enough for one focused run.
RRelevant: linked to the spec, bug, decision, or user value.
TTime-bounded: constrained by run budget, retry limit, or slice size.

Inside Sandman

The autonomous delivery workflow, from `sandman run 1234` to merged PR.

The user usually selects the issue. Sandman owns what happens after the command: it keeps the agent moving through concrete delivery gates while preserving traditional engineering controls.

Sandman-owned workflow$ sandman run 1234
User starts

Select one ready issue

You choose the work explicitly, usually with `sandman run 1234`.

01

Create isolated sandbox

Worktree or container, with the rendered task and repo context.

02

Plan implementation

Turn the issue into a repo-grounded execution plan before editing.

03

Build with feedback

Use TDD, targeted tests, type checks, and repair loops where possible.

04

Self-review and open PR

Check standards and issue fidelity, then create the pull request.

05

Drive review to merge

Request review, apply feedback, back-merge if needed, require green checks, then merge.

Selection is explicit by default.

Sandman can run ranges, labels, or frontiers, but the simplest story is one ready issue selected by the developer.

The output is a merged PR.

Sandman is not just a code generator or reviewer. It owns the delivery path until the pull request is merged.

Every run leaves receipts.

Logs, event state, branches, task prompts, PR links, review requests, CI outcomes, and merge decisions are visible after the fact. Sandman is designed for AFK work you can audit, not agent output you have to trust blindly.

Methodology, correctly placed

Use any upstream method. Sandman is the delivery loop.

Matt Pocock skills, SDD, and Loop Engineering are useful concepts, but they are not Sandman's dependency chain. They occupy different layers around the same goal: making agent work more reliable.

LayerPrecise roleSandman's relationship
Matt Pocock skillsComposable engineering skills for alignment, domain language, specs, tickets, TDD, and review.Optional upstream helper. v1 used `to-prd` / `to-issues`; v1.1 uses `to-spec` / `to-tickets`.
Spec-driven developmentSpecs become the source of truth that drives plans and tasks.Optional upstream source of agent-ready issues. Sandman does not need to own the spec layer.
Loop EngineeringThe broader discipline of recurring agent systems with triggers, state, verification, budgets, and exits.Sandman is a concrete delivery loop for GitHub issues: issue intake to merged PR.

The category is not “AI coding assistant.”

Sandman is closer to an autonomous delivery system. It does the repetitive implementation loop that normally forces you to keep prompting the agent: plan the issue, code it, verify it, respond to review, and merge it.

The upstream method can change. The handoff stays the same: SMART GitHub issue -> Sandman -> reviewed, merged PR.
