What it is

A practical harness for agentic software development.

Sandman is for developers who already know that coding agents can do useful work, but do not want to babysit every shell command, branch, pull request, review request, and retry. It gives agents a place to work and gives developers a way to inspect what happened.

1

Pick work that is ready

Use a single issue, a label query, a prompt, or a prepared spec. Clear acceptance criteria beat vague prompting.

2

Run it in isolation

Sandman creates a worktree or container so the agent does not touch your active working tree.

3

Follow workflow rails

Planning, TDD, implementation, self-review, back-merge, PR review, and merge gates are encoded as skills.

4

Review real artifacts

Use status, history, attach, portal, logs, events, branches, commits, and PRs instead of trusting a chat transcript.

Recommended workflows

Use Sandman after intent is clear, before execution gets tedious.

The behavior-first workflow

Use planning and issue-shaping skills to turn an idea into agent-ready work. The language from real sessions is direct: grill the plan, create tracer-bullet slices, define testability seams, then hand execution to Sandman.

/to-prd
/to-issues
/grill-me
sandman run label:agent-ready --parallel 4

The PR review workflow

Sandman shines after the first PR exists. The review loop waits for CI, requests delegated review, polls feedback, applies changes, pushes, and repeats until approved or exhausted.

sandman-pr-review

wait for checks
post /sandman review
apply feedback
repeat until approved

The batch workflow

Queue multiple issues, respect blockers, and let independent tasks run at once. Failed blockers skip dependents instead of creating confused follow-on work.

sandman run 42:48 --include-dependencies --parallel 4

The recovery workflow

When a run fails, you do not start from nothing. Continue on the existing branch, inspect logs, attach to the run, or use history to understand the last durable state.

sandman status
sandman attach 42
sandman run --continue 42
The developer stays responsible for judgment. Sandman takes responsibility for the boring loop.

Compare development modes

From hand-driven development to supervised agent execution.

Mode Developer experience Failure mode What Sandman changes
Traditional You manually plan, branch, code, test, push, request review, and merge. High quality, but routine work competes with deep work. Preserves review culture while moving routine execution into background runs.
Prompt-driven agent work You ask an assistant to modify code in the current context. Fast, but decisions and commands are hard to audit later. Creates durable runs with branch, logs, task prompts, events, and review state.
Spec-driven development You specify what and why before implementation starts. Better intent, but implementation still needs orchestration. Turns specs or task lists into sandboxed runs and mergeable slices.
Cloud agent platforms You delegate work to a hosted agent product. Convenient, but workflow control belongs to the platform. Keeps orchestration local, CLI-native, explicit, and adaptable.

Current competitors

Sandman sits beside the agents, not in front of them.

Devin, Copilot cloud agent, Codex, Jules, Cursor, Kiro, Factory, CodeRabbit, and Spec Kit all prove the market is moving toward asynchronous agentic development. Sandman differentiates by focusing narrowly on the developer-owned execution harness: isolation, batches, workflow skills, status, review loops, and merge gates.

A

Against Devin or Jules

Do not buy a black-box worker when you mainly need orchestration around your own repo, agent, and workflow.

B

Against Copilot or Codex

Those are powerful agents and surfaces. Sandman is the process harness that can wrap agent work.

C

Against Kiro or Spec Kit

Specs are excellent inputs. Sandman is the execution and review system after the spec is ready.

D

Against review bots

Review bots inspect changes. Sandman coordinates the agent that creates, fixes, reviews, and merges them.
