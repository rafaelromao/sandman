The handoff contract

Sandman starts after intent is clear enough to delegate.

Sandman is workflow-agnostic. A staff engineer, SDD flow, Matt Pocock skills, bug diagnosis session, or your own planning ritual can produce the issue. Sandman only needs a well-described GitHub issue and a repository it can work in.

Agent-ready issues are SMART enough to run while you are away.

SMART is the minimum viable contract between human intent and an AFK coding agent.

SSpecific: behavior, scope, and non-goals are concrete.
MMeasurable: acceptance criteria or checks define done.
AAchievable: small enough for one focused run.
RRelevant: tied to a spec, bug, architecture decision, or user value.
TTime-bounded: constrained by run budget, retry limit, or slice size.
before

Write the spec and slice the work.

Use the upstream method you trust. The important output is a GitHub issue with clear behavior and verification.

sandman

Run the AFK delivery loop.

Sandman plans, sandboxes, builds, tests, self-reviews, opens the PR, drives review, applies feedback, and merges.

after

Validate the reference branch.

The merge is not the end of engineering judgment. Treat it like human work: smoke it, run broader tests, QA it, then promote.

From spec to validated branch, with Sandman owning the middle.

The flow makes the boundary explicit. Humans and upstream tools shape intent before Sandman. Sandman owns implementation through merged PR. Your validation practice proves the merged reference branch is ready for the next environment.

The middle is Sandman-owned.

Sandman is not a code generator. It is the delivery loop from selected issue to reviewed, merged PR.

Review is the new bottleneck.

AI makes code cheap. Sandman makes review and verification explicit gates instead of afterthoughts.

Reference branch is not production.

Merge into a branch you can validate. Production promotion should remain a deliberate release step.

Operate the loop

Autonomous work still needs visible state.

Sandman gives you receipts instead of vibes: run state, task files, logs, branches, PR links, review turns, CI outcomes, and merge decisions. You do not babysit the agent, but you can inspect the loop at any time.

After merge

Sandman should feed validation, not bypass it.

Treat Sandman output the way you would treat fast human output. The implementation loop can merge to a reference branch, but release confidence comes from recurring validation across multiple changes.

Good engineering still applies.

TDD is not an AI novelty. It is a traditional control that helps agents work in smaller safe loops. Keep it, then add broader validation where the risk demands it.

  • OKTargeted tests during implementation.
  • OKSelf-review before PR creation.
  • OKGreen CI before merge.
  • OKSmoke, e2e, QA, or wider suites after groups of merged work.

Current dependencies are explicit.

Today, Sandman expects GitHub for source control and issue workflow, and OpenCode as the implementation agent. That is a current integration boundary, not the eternal product category.

The durable idea is workflow-agnostic AFK delivery: any process can feed Sandman clear issues, and Sandman can evolve to support more agents and forges later.

Methodology, correctly placed

Use the upstream method that creates the clearest issues.

Sandman should lead the marketing story because it owns the autonomous delivery loop. Other skills and disciplines help around it: before the run to create better work, and between runs to keep the codebase healthy.
