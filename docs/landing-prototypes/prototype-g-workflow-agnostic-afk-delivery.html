Product boundary

Sandman starts after the issue is ready.

The clearest way to explain Sandman is by separating responsibility. Other workflows can discover, debate, specify, and slice work. Sandman replaces the AFK implementation phase once there is a well-described GitHub issue to execute.

Before Sandman

Create the work item however your team likes.

  • Matt Pocock v1: `grill-with-docs -> to-prd -> to-issues`.
  • Matt Pocock v1.1: `grill-with-docs` or `wayfinder -> to-spec -> to-tickets`.
  • Spec-driven development, Kiro, Spec Kit, Linear, Jira, or a human-written issue.

Sandman

Run the implementation loop while you are away.

  • Select ready GitHub issues and respect blockers.
  • Create isolated worktrees or containers.
  • Drive plan, TDD, self-review, PR creation, PR review, and merge.
  • Show progress in Sandman Portal.

After Sandman

You return to auditable delivery artifacts.

  • Merged pull requests, not loose patches.
  • Logs, run state, task prompts, branches, review history, and issue links.
  • Human oversight where product or architecture judgment is still needed.
The input contract

Agent-ready means SMART enough to run AFK.

Sandman is workflow-agnostic, but it is not ambiguity-proof. The handoff works best when GitHub issues are written as SMART implementation slices: specific, measurable, achievable, relevant, and time-bounded enough for one agent run.

S

Specific

The issue names the concrete behavior to change, the user or operator outcome, and what is out of scope.

M

Measurable

Acceptance criteria, tests, commands, screenshots, or review checks make completion observable.

A

Achievable

The slice is small enough for a fresh context window and does not require unresolved product judgment.

R

Relevant

The issue links back to the parent spec, PRD, bug, business goal, or technical decision it advances.

T

Time-bounded

For Sandman, this means bounded by one run or a clear retry budget, not necessarily a calendar deadline.

Workflow agnostic by design

Bring your own way of creating the issues.

Sandman is not a spec tool, a methodology, or a replacement for product thinking. It is the AFK delivery harness that begins when your issue tracker has the next ready frontier.

Matt Pocock skills v1

Use `grill-with-docs`, `to-prd`, and `to-issues` if that is your current muscle memory. Sandman does not care about the names. It cares that GitHub issues are ready.

Matt Pocock skills v1.1

Use `grill-with-docs` or `wayfinder`, then `to-spec` and `to-tickets`. Sandman runs the ticket frontier after publication.

Spec-driven development

Use Spec Kit, Kiro, or another SDD process to define intent and tasks. Sandman executes the resulting GitHub issues.

Your own process

Use PM-written issues, tech-lead breakdowns, Jira-to-GitHub sync, or hand-authored tasks. If they are SMART, Sandman can run them.

What Sandman owns

AFK implementation, from planning to merge.

The Sandman promise is not “an agent wrote code.” The promise is a completed delivery loop: implementation plan, tested changes, self-review, pull request, review loop, and merge gate, with monitoring throughout.

01

Select issue

Pick ready GitHub issues, ranges, labels, queries, or the current unblocked frontier.

02

Sandbox

Create an isolated worktree or container so the agent does not touch your checkout.

03

Plan

Turn the issue into an implementation plan grounded in the repo.

04

TDD

Use tests and validation loops where possible before broad implementation.

05

Self-review

Review against standards and the originating issue/spec before asking others.

06

PR review

Use `sandman review` or any review agent that responds to PR comments.

07

Merge

Back-merge if needed, require green checks and approval, then merge the PR.

The sharp one-liner

Write the issues your way. Let Sandman merge them while you sleep.

Sandman is the workflow-agnostic AFK implementation layer for teams that already know how to describe work. Give it SMART GitHub issues. It gives you merged PRs.
