The workflow you actually use

From foggy idea to merged PR fleet.

Sandman is not the whole process. It is the AFK execution half of a larger engineering loop: clarify the work like you would with humans, publish the spec, break it into slices, then let agents execute those slices in isolated sandboxes.

1

grill-with-docs or wayfinder

Resolve uncertainty, sharpen vocabulary, write ADRs and domain terms.

2

to-spec

Publish the PRD/spec: problem, solution, user stories, decisions, testing seams.

3

to-tickets

Break the spec into tracer-bullet GitHub issues with blocking edges.

4

sandman run

Run ready issues in worktree or container sandboxes, often in parallel.

5

plan + TDD

The agent plans the issue and drives red-green work at agreed seams.

6

self-review + PR

Review against standards and spec, then open a durable pull request.

7

review daemon

`sandman review`, OpenCode, Codex, or another review agent responds to PR comments.

8

merge gate + Portal

Monitor progress, require green checks and approval, then merge.

The promise

Do the thinking awake. Let Sandman run the loop asleep.

A good spec tells the agent what matters. Good tickets make the work sliceable. Sandman makes the execution observable, isolated, reviewable, and mergeable while you are away.
