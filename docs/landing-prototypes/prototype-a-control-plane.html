Sandman is not another coding agent.

It is the operating layer around the agents you already use. The agent writes code. Sandman gives that work a sandbox, a task file, a status trail, a review loop, and a merge discipline.

What it is

A CLI that turns prompts, issues, or prepared implementation plans into isolated agent runs. Each run writes logs and events under `.sandman/` so progress can be inspected instead of guessed.

What it is for

Routine backlog work, test generation, refactors, docs, bug fixes, review iterations, and any issue that is clear enough to hand to an agent while you go AFK.

How it works

Sandman selects work, creates a worktree or container sandbox, renders the task prompt, runs the configured agent, tracks events, exposes logs, and keeps the run inside workflow rails.

The recommended workflow is the product.

Sandman was built around the pattern that kept recurring in real sessions: shape the intent, plan behavior-first, implement a tracer bullet, review against standards and spec, then ask the PR review agent to keep pushing until approval.

Use it like a senior engineer delegates work.

Sandman works best when you give agents clear boundaries and then let the workflow run without babysitting.

For one issue

Run a single tracked task in its own branch, inspect the portal, and pick up the PR when the review loop is done.

sandman run 42

For a backlog slice

Run independent issues in parallel while blockers wait their turn. Come back to logs, branches, and PRs rather than chat scrollback.

sandman run label:agent-ready --parallel 4

For prepared specs

Use spec-driven tools or planning skills to create intent, then let Sandman execute each implementation slice in isolation.

sandman run --prompt "Implement the next task"

Traditional development keeps judgment. Sandman removes waiting.

Sandman does not replace human engineering judgment. It removes the serial glue work around routine implementation: branch setup, task prompting, retries, logs, review requests, and merge gates.

Approach What it gives you What still hurts Sandman's role
Traditional software development Human judgment, mature review culture, traceability. Routine work is slow, serial, and interruption-heavy. Keep the judgment. Delegate the execution.
IDE assistants Fast synchronous pair-programming. The developer still drives branch, PR, review, CI, and merge steps. Move agent work out of the editor and into isolated background runs.
Spec-driven development Clearer intent, better planning, less vibe coding. Specs still need disciplined execution and review. Use specs as input, then execute them through sandboxes and gates.
Cloud coding agents Background PR creation. Vendor workflow, limited process control, another place to govern. Provide your own CLI-native operating model around agent work.

Competitors give you agents. Sandman gives you rails.

Devin, Copilot cloud agent, Codex, Jules, Cursor, Kiro, and Factory are moving coding into the background. Sandman focuses on the missing layer: how to contain, observe, review, and merge the work reliably.

Agent agnostic

Start with OpenCode today. The product idea is broader: do not bet your engineering process on one vendor's agent surface.

Local and explicit

Sandman uses repo-visible prompts, worktrees, logs, events, and review gates. The process is inspectable.

AFK by design

The workflow assumes the developer is away. Agents must proceed, recover when possible, and leave auditable work behind.

"The job is not to make agents more magical. The job is to make their work boring enough to trust."
