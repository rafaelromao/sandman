Software engineering fundamentals for agents

AI did not make engineering discipline obsolete.

Sandman works because it treats agents like fast junior engineers, not magic. Give them a spec, small tickets, isolated branches, tests, review, CI, and merge gates.

The method

The same delivery controls, adapted for AFK agents.

Traditional software development already solved most of the trust problems that agentic coding reintroduces: ambiguous requirements, oversized tasks, hidden work, missing tests, weak review, and unsafe merging. Sandman packages those controls around coding agents.

Spec

Context before code

The PRD/spec explains what we are implementing and why, so every issue has broader context.

Tickets

Tracer-bullet slices

Each GitHub issue is a small implementation slice with acceptance criteria and blocking edges.

Sandbox

One worker, one workspace

Agents get isolated worktrees or containers instead of editing the developer's checkout.

TDD

Feedback while building

Sandman drives issue implementation through planning and test-first slices where possible.

Review

Standards and spec

Self-review and PR review check both engineering standards and fidelity to the originating issue/spec.

Merge

Green and approved

PRs move through CI, delegated review, back-merge, and approval before the merge gate opens.

Human team analogy

Sandman manages agent work like a staff engineer manages a team.

With human engineers

  1. Discuss the feature until requirements are clear.
  2. Write the spec or PRD.
  3. Break it into tickets.
  4. Assign independent slices.
  5. Review PRs against the spec and standards.
  6. Merge only when checks and review pass.

With Sandman agents

  1. Use `grill-with-docs` or `wayfinder` to resolve uncertainty.
  2. Use `to-spec` to publish the PRD/spec.
  3. Use `to-tickets` to create GitHub issues.
  4. Run the ready frontier with `sandman run`.
  5. Use Sandman self-review and review agents on PRs.
  6. Merge through Sandman's gate.
